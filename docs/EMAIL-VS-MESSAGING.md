# «Почта» в документации: это не личные сообщения

## Что имелось в виду под «почтой в dev»

Речь шла о **транзакционной электронной почте (SMTP)** — письмах **с сервера наружу** (или в тестовый ящик), которые Django отправляет для сценариев вроде:

- **восстановление пароля** — ссылка с `uid` и токеном на ваш фронт;
- при желании позже: подтверждение email, уведомления «вам написали» (опционально, не путать с доставкой ЛС).

**Личные сообщения (ЛС)** в приложении — это **ваш API + WebSocket + PostgreSQL**, отдельный канал. Они **не проходят** через SMTP и **не** «перехватываются Mailhog’ом».

## Mailhog в проекте (зафиксировано)

В **docker-compose** для разработки поднимается сервис **Mailhog**: Django направляет **SMTP** на `mailhog:1025`, письма смотрите в UI (например `http://localhost:8025`). Это стандартный способ тестировать **сброс пароля** без реальной почты.

**Консольный email backend** (письма в логи) можно оставить как запасной режим через env, но **основной dev-поток** — Mailhog в compose. См. [DECISIONS.md](./DECISIONS.md).

Оба подхода — **только для разработки**. В production письма идут через реальный SMTP (SendGrid, SES, Mailgun и т.д.) или API провайдера.

## Свой почтовый домен — сложно?

**Средняя сложность**, не «rocket science»:

1. Купить/привязать домен, завести **SPF**, **DKIM**, иногда **DMARC**.  
2. Не слать спам, следить за репутацией IP (у VPS сложнее, у transactional-провайдера проще).  
3. Для MVP чаще берут **готовый transactional API** — домен привязывают для «From:», DNS настраивают по инструкции провайдера.

Это **не обязательно** на старте, если сброс пароля в dev идёт через **Mailhog** в compose.

## Можно ли «перехватывать» ЛС Docker’ом, как пакеты?

- **ЛС по HTTP/WebSocket** между браузером и API — это **обычный сетевой трафик** к вашему хосту/контейнеру.  
- **Docker** сам по себе не показывает «письма ЛС» — он оркестрирует контейнеры. Смотреть трафик можно так:
  - **DevTools** в браузере (вкладки Network, WS) — лучший вариант для разработки;
  - **mitmproxy** / Charles — если нужен детальный разбор HTTPS (с доверием к сертификату);
  - **`docker compose logs`** — только то, что приложение **само** залогировало (в т.ч. если залогировать входящие WS-сообщения на бэке — осторожно с приватностью).

Итого: **перехват ЛС ≠ почта**. Для отладки ЛС используйте **WebSocket frame** в DevTools или логи на сервере с флагом debug.
