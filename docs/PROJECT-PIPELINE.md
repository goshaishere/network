# Пайплайн проекта: от идеи до релиза

Документ задаёт **порядок фаз** и критерии готовности. Нумерация сквозная; фичи из более поздних фаз не подтягивают в ранние без явной необходимости.

**Снимок «док ↔ код»** (чтобы статусы не расходились): **[IMPLEMENTATION-STATUS.md](./IMPLEMENTATION-STATUS.md)** — обновлять при заметных изменениях в `backend/` / `frontend/` или при закрытии подфазы.

## Фаза 0 — Документация

- [x] Архитектура, датафлоу, JWT, Mailhog, Redis, WS, тесты, CI — см. остальные `docs/`.  
- [x] Два контура, **сообщества** / **рабочие группы**, **internal/partner**, разведение **`/`** / **`/dashboard`** / **`/work`**, единый перечень **semantic** — [ROLES-AND-TASKS.md](./ROLES-AND-TASKS.md), [DECISIONS.md](./DECISIONS.md).  
- [x] CI/CD: **GitHub Actions** по умолчанию (см. [CI-CD.md](./CI-CD.md)); при другом хостинге — перенос job’ов.

**Выход:** актуальные `docs/*.md`.

## Фаза 1 — Каркас репозитория

- Django: split settings, DRF, **SimpleJWT**, CORS, **ASGI** (Daphne/Uvicorn), **Channels**, **Redis**.  
- Quasar: лейауты, роутер, i18n, тёмная тема.  
- `docker-compose`: Postgres, Redis, Mailhog, API.  
- CI: ruff + pytest (Postgres); ESLint + vue-tsc + Vitest + build. См. [CI-CD.md](./CI-CD.md), [TESTING.md](./TESTING.md).

**Критерий:** `compose up`, зелёный CI на PR.

---

## Фаза 2 — Аутентификация и безопасность входа

- JWT, refresh/blacklist, `me`, сброс пароля (Mailhog в dev), rate limit + hCaptcha с 3-й попытки.  
- Страницы auth + 404.

**Критерий:** сценарии входа/сброса стабильны; нет утечки «существует ли email».

---

## Фаза 3 — Профиль, стена, главная и дашборд

- Profile, Post, страница пользователя, настройки.  
- **Лёгкая главная** `/` (или `/feed`) — фиксированные блоки + ссылка на дашборд.  
- **`/dashboard`** — каталог виджетов, добавление/удаление, **закрепление**, сохранение раскладки (`GET/PATCH .../me/dashboard/`), режим редактирования. См. [ROLES-AND-TASKS.md](./ROLES-AND-TASKS.md) §5.  
- Настройка **«Стартовая страница»** после логина (опционально в этой же фазе).

**Критерий:** посты на стене; пользователь может собрать дашборд и перезайти — раскладка сохраняется.

---

## Фаза 4 — Личные сообщения (MVP)

- REST + **WebSocket** для доставки сообщений.

**Критерий:** диалог в реальном времени без ручного обновления.

---

## Фаза 5 — Сообщества (соцконтур, для всех)

- Приложение **`communities`**: создание, вступление, посты сообщества, открытое/закрытое.  
- Фронт: **`/communities`**, карточка сообщества — доступ **всем** ролям **`user`** и выше (гостю — только публичные экраны по политике).

**Критерий:** пользователь без `employee` создаёт/вступает в сообщество и пишет на стене сообщества; **`/work/*` по-прежнему недоступен**.

## Фаза 6 — Продакшен и наблюдаемость

- [x] Статика, env prod, **`/api/v1/health/`** (БД, Redis, диск), **`/health/live/`**, **`/health/ready/`**, деплой, бэкапы БД.  
- [x] Логи (`LOG_LEVEL`), uptime; **Sentry** + **Prometheus** (`/metrics/`, токен scrape) + опционально **Grafana** — [MONITORING-AND-SUPPORT.md](./MONITORING-AND-SUPPORT.md), [runbooks/observability.md](./runbooks/observability.md).

**Критерий:** healthcheck и минимальный мониторинг/алерты — выполнено.

---

## Фаза 7 — Роли сотрудника: штат vs партнёр

- [x] Роли **`user` / `employee` / `admin`**; поле **`EmploymentKind`** (`internal` | `partner`), выдача типа штат/партнёр из админки.  
- [ ] Опционально: **`Tenant`** — вне текущего MVP.  
- [x] API: префикс **`/api/v1/internal/...`** только для **`internal`** (или staff); общий рабочий API для всех сотрудников.  
- [x] Фронт: скрытие внутренних пунктов меню и запросов к internal API для `partner`; на **`/work`** — различие подсказок/виджетов по `employment_kind`.

**Критерий:** партнёр не получает 200 с internal-only эндпоинтов; штат видит расширенные сводки на **`/work`** через **`/internal/work/dashboard/`**; дальнейшие виджеты канбана — по мере развития фазы 8.

---

## Фаза 8 — Рабочий хаб, рабочие группы, канбан

- [x] **`WorkGroup`**, **`WorkGroupMembership`**, доски, колонки с **семантикой** и пресетами **`generic_pm` / `it_sdlc` / `custom`** — [ROLES-AND-TASKS.md](./ROLES-AND-TASKS.md) **§6–7**.  
- [x] Главная работы **`/work`** → **`/work/groups`**, **`/work/groups/:id`**; агрегаты **`GET /work/dashboard/`** (группы, доски, задачи с дедлайнами).  
- [x] Задел под **лёгкий CRM:** модели **`WorkCounterparty`**, **`WorkContact`** (без связи с задачами в MVP).  
- [ ] **WS** на доске, автоматизации, расширенные фильтры — вне закрытия фазы 8.

**Критерий:** только **`employee`+**; участник рабочей группы видит доски и данные; неучастник не получает чужие колонки/задачи и **403** на создание доски в чужой группе; **`user`** получает **403** на **`/work/*`** и на API **`/tasks/...`**.

## Фаза 9 — Панель администрирования

- [x] `/console`: пользователи (**`PATCH /admin/users/`** — `is_employee`, `employment_kind`, отдел, **`permission_group_ids`**), роли, **`EmploymentKind`**, компании/отделы (**`POST`** организаций и отделов).  
- [x] **Группы пользователей** (`UserPermissionGroup`): CRUD, **`member_ids`**, **`permission_slugs`**; каталог **`GET /admin/permission-catalog/`**; **`effective_permission_slugs`** в **`GET /auth/me/`**.  
- [x] **Модерация сообществ из консоли:** **`GET /admin/communities/`**, **`PATCH /admin/communities/<id>/`**, **`GET .../<id>/posts/`**, **`DELETE /admin/communities/posts/<id>/`** (только staff/`is_superuser` через **`IsAdminUser`**).  
- [ ] Тонкая проверка **каждого** permission slug на всех эндпоинтах приложения — поэтапно после каталога.

**Критерий:** админ назначает сотрудника и тип штат/партнёр без правки БД; может **создавать группы**, **включать в них пользователей** и **выдавать группам различные разрешения** — из `/console`; модерация сообществ и постов — через те же API консоли.

## Фаза 10+ — Соцрасширение «как ВК»

- [x] Друзья (заявки, список) и **персональная лента** (`GET /social/feed/`, бесконечный скролл на фронте).  
- [x] **Моя страница** `/me` (профиль + своя стена), **мои сообщества** `GET /communities/mine/`, навигация: лента, друзья, сообщения, группы.  
- [ ] Вложения в ленту, пуши, полноценная модерация ленты — вне текущего инкремента.

## Фаза 11 — Контейнеризация и CI/CD

- Укрепление `docker-compose` для dev/stage/prod профилей, healthchecks, корректный startup-order и переменные окружения.  
- Dockerfile(ы) для backend/frontend (multi-stage), оптимизация образов и кэширования слоёв.  
- CI/CD: сборка и публикация образов, миграции при деплое, smoke/health-check после выката, правила rollback.  
- Проверка секретов и безопасная конфигурация pipeline (без хранения чувствительных данных в репозитории).

**Критерий:** reproducible build и deploy из CI; сервисы поднимаются контейнерами стабильно, деплой проходит автоматически с проверкой здоровья.

## Статусы задач — что зафиксировано в доке

Источник правды: **[ROLES-AND-TASKS.md §7](./ROLES-AND-TASKS.md)**.

| Правило | Содержание |
|---------|------------|
| Что такое «статус» | Позиция задачи = колонка **`Column`** на доске **`Board`**. |
| Семантика | Поле **`semantic`** — из **единого списка** (не путать с отображаемым названием колонки). |
| Успешное закрытие | В отчётах: `semantic in ('done', 'released')`. |
| Отмена | Терминал неуспеха: `cancelled`. |
| Пресеты только при создании | **`generic_pm`** — 6 колонок (`planned` → … → `cancelled`); **`it_sdlc`** — 4 (`backlog` → `released`); **`custom`** — пусто/минимум. |
| Расширение | После создания доски колонки можно **добавлять** с любой `semantic` из §7.1 или без неё (`null`). |

При реализации: валидировать `semantic` на бэке при `POST/PATCH` колонок; сид пресета — один сервис (`tasks/services/columns.py`).

## Окончательный порядок при старте кода

**Жёсткая последовательность (зависимости):**

1. **Фаза 1** — без неё нет ни одной фичи.  
2. **Фаза 2** — auth обязателен для остального UI.  
3. **Фазы 3–5** — можно идти **3 → 4 → 5**; альтернатива для быстрого соц-MVP: **3 → 5 → 4**, если ЛС отложить (не рекомендуется, если WS уже в стеке).  
4. **Фаза 6** — для публичного продакшена; для **внутреннего стенда** можно поднять упрощённо после **фазы 2**.  
5. **Фазы 7 → 8 → 9** — **7 перед 8**, если сразу нужны различия **internal/partner** на рабочих API и виджетах; если первый спринт работы только для штата — допустим черновик **8** без `EmploymentKind`, затем **7** как усиление. **Группы пользователей и матрица прав по группам** (фаза **9**) опираются на стабильные роли из **7** и точки интеграции в API; при необходимости — после появления целевых эндпоинтов в **8**.  
6. **Фаза 10+** — по продуктовому приоритету.

**Первый «полезный» инкремент для демо:** конец **фазы 3** (профиль + стена + `/` + `/dashboard`) или конец **фазы 5** (ещё и сообщества без работы).

**Первый полный продукт «соц + работа»:** фазы **1–9**; экстра — почта, тяжёлый CRM, WS на доске.
