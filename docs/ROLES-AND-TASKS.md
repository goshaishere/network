# Роли, сообщества, рабочий контур, статусы задач и админка

Документ описывает **два разделённых контура** продукта **Network** и то, как они сочетаются.

| Контур | Кто видит | Суть |
|--------|-----------|------|
| **Социальный** | Все авторизованные (**`user`** и выше) | Профиль, стена, ЛС, **сообщества** (как публичные/закрытые группы ВК). |
| **Рабочий** | Только **`employee`** и **`admin`** | Рабочие группы, доски, канбан, «дом работы», внутренние инструменты по типу сотрудника. |

> **`django.contrib.auth.Group`** не использовать как имя сущности «сообщество» / «рабочая группа». В коде: **`Community`**, **`WorkGroup`**.

---

## 1. Роли (`user` / `employee` / `admin`)

| Роль | Соцконтур | Рабочий контур `/work/*` |
|------|------------|---------------------------|
| **`user`** | Да: лента (позже), сообщества, профиль, ЛС | **Нет** (API → 403). |
| **`employee`** | Да | Да: группы, доски, задачи, главная работы. |
| **`admin`** | Да (можно сузить политикой) | Да + **консоль** `/console/*`. |

Проверки на **бэке** обязательны; меню и редиректы на фронте — дополнение.

---

## 2. Профиль (соц, сотрудник, дашборд)

- **Общее:** био, аватар, приватность, локаль.  
- **Для сотрудника (`employee+`):** компания, отдел, должность (справочники `directory`).  
- **Для `/dashboard`:** **`dashboard_layout`** (JSON раскладки виджетов, см. §5.2).  
- **`default_landing`** (или `UserPreference`) — `home` | `dashboard` | `work` (если роль позволяет).

---

## 3. Сообщества (`Community`) — для всех

Отдельное приложение **`communities`** (или раздел в `social`):

- Создание/вступление, роли в сообществе (создатель, модератор, участник).  
- Стена/посты сообщества, обложка, тип **открытое / закрытое**.  
- **Не** путать с **`WorkGroup`**: сообщество — про интересы и общение; рабочая группа — про задачи и NDA-контур заказчика.

**API (черновик):** `/api/v1/communities/`, `.../members/`, `.../posts/`.

**Фронт:** `/communities`, `/communities/:slug` — доступны **всем** с ролью не ниже зарегистрированного пользователя (гостю — только публичные страницы по политике).

---

## 4. Сотрудник: «свой» (штат) vs партнёр и сценарий заказчика

### 4.1. Зачем различать

Продукт могут **разворачивать как заказчик** (одна организация) или вести как **SaaS** (несколько организаций). Нужно:

- дать **штату** доступ к **внутренним** сущностям и шаблонам;  
- ограничить **партнёра** (клиент, подрядчик «поработать») рамками **выданных** рабочих групп и досок, без лишних внутренних разделов.

### 4.2. Модель (рекомендация)

| Поле / сущность | Назначение |
|-----------------|------------|
| **`Tenant`** (опционально в MVP) | Организация-заказчик инсталляции; в single-tenant один дефолтный tenant. В SaaS — у каждого клиента свой. |
| **`EmploymentKind`** на профиле сотрудника | `internal` — штат организации владельца инсталляции; `partner` — внешний участник (клиент, аутстафф и т.д.). |
| **`partner_organization`** (FK, nullable) | Юр./логическая связь партнёра с контрагентом для CRM. |
| **Флаги возможностей** | Не хардкодить везде `if partner`: лучше **`FeatureSet`** / permissions: `can_see_internal_modules`, `can_manage_tenant_settings`. |

**`internal`:** полный рабочий дашборд, внутренние отчёты, шаблоны досок организации, будущие модули «только штат».  
**`partner`:** только явно выданные **WorkGroup**, свои задачи, без разделов внутреннего HR/вики заказчика (если такие появятся).

### 4.3. Кто назначает

Только **`admin`** (или интеграция HR) выставляет `EmploymentKind` и доступы; саморегистрация по умолчанию даёт **`user`**, не `employee`.

## 5. Главная страница, настраиваемый дашборд и рабочий хаб

Здесь разведены три разные «точки входа», чтобы не смешивать **лёгкую главную**, **персональную конструкторскую** страницу и **рабочий контур**.

### 5.1. Что где (коротко)

| Маршрут | Назначение | Кто |
|---------|------------|-----|
| **`/`** или **`/feed`** | **Соцглавная (лёгкая):** фиксированный небольшой набор блоков (приветствие, 1–2 рекомендации, быстрые ссылки на «Сообщества», «Сообщения», **«Мой дашборд»**). **Не** сюда тащить тяжёлый конструктор виджетов — страница должна быстро открываться и быть понятной новичку. | Все авторизованные |
| **`/dashboard`** | **Персональный настраиваемый дашборд:** сетка виджетов, своя раскладка, см. §5.2. | Все авторизованные |
| **`/work`** | **Рабочий хаб** (только `employee`+): ориентир на задачи, группы, дедлайны, внутренние блоки для `internal`. Может **дублировать данные** тех же сущностей, что и виджеты на `/dashboard`, но в фиксированном «рабочем» UX. | `employee`, `admin` |

**После логина:** по умолчанию **`/`** (предсказуемо); в настройках профиля — опция **«Стартовая страница»**: `home` | `dashboard` | `work` (если роль позволяет). Сохранять в `Profile` или `UserPreference`.

### 5.2. Настраиваемый дашборд (`/dashboard`)

**Идея:** пользователь сам собирает экран из **виджетов** из каталога; раскладка и состав сохраняются на сервере (и при желании кэшируются в Pinia).

**Каталог виджетов (примеры):**

| `widget_type` | Содержимое | Доступность |
|---------------|------------|-------------|
| `communities_summary` | Мои сообщества, счётчики | все |
| `messages_preview` | Последние диалоги / непрочитанное | все |
| `feed_snippet` | Лента (когда появится) | все |
| `tasks_due` | Мои задачи с ближайшим сроком | только **`employee`+** (для `user` не показывать в каталоге или показывать заглушку «станьте сотрудником») |
| `work_groups_shortcut` | Быстрый список рабочих групп | `employee+` |
| `internal_news` | Внутренние объявления | только **`internal`** |

**Действия пользователя:**

- **Добавить** виджет из боковой панели / модалки «Каталог».  
- **Убрать** — с подтверждением, если виджет **закреплён** (см. ниже).  
- **Перетаскивание** (drag-and-drop) и изменение размера в сетке — библиотека уровня **`vue-grid-layout`** или CSS **Grid** + сохранённые `grid-area` / `row`/`col` (проще для MVP).  
- **Закрепление (`pinned`):**  
  - в **логике:** виджет с `pinned: true` не удаляется без явного «Открепить и удалить»; при добавлении новых **закреплённые** остаются в верхней зоне (отдельный ряд «Закреплёно») **или** всегда в начале списка в порядке чтения;  
  - в **UI (опционально):** для 1–2 виджетов — режим **«липкий»** (`position: sticky` внутри `QPage`), чтобы при прокрутке длинной сетки блок оставался видимым (осторожно на мобильных — ограничить высоту).

**Модель хранения (рекомендация):**

- JSON в **`User.dashboard_layout`** или отдельная таблица **`DashboardWidgetInstance`**: `user`, `type`, `pinned`, `x`, `y`, `w`, `h`, `config` (произвольные props), `version` схемы для миграций.  
- **`GET/PATCH /api/v1/profiles/me/dashboard/`** — отдать/сохранить layout; валидация на бэке: только **разрешённые** `widget_type` для роли и `employment_kind`.

**Фронт:** страница **`DashboardPage.vue`**, режим «Редактирование» (переключатель), composable **`useDashboardLayout`**.

### 5.3. «CRM + Jira» для заказчика (дорожная карта)

Не всё в MVP.

| Модуль | Зачем | Примечание |
|--------|--------|------------|
| **Задачи + канбан** | Как Jira/Trello | §6–7. |
| **Контакты / организации (лёгкий CRM)** | Контрагенты, связь с задачами | `Organization`, `Contact`. |
| **Воронка / сделки** | Pipeline продаж | после MVP. |
| **Календарь / сроки** | Дедлайны | данные те же, что у виджета `tasks_due` на `/dashboard` и `/work`. |
| **Активность по задаче** | История в карточке | лента событий. |
| **Внутренние модули** | Только `internal` | префикс `/internal/`. |

Итого: **`/`** — простая соцточка; **`/dashboard`** — «мой экран под себя»; **`/work`** — фокус на работе для сотрудников.

## 6. Рабочие группы (`WorkGroup`) — только сотрудники

Как в предыдущей версии: команда вокруг досок, **не** сообщество.

- `WorkGroup`, `WorkGroupMembership`, `Board.work_group`.  
- Доступ: участник группы + политика для **`partner`** (только выданные группы).  
- API: `/api/v1/tasks/work-groups/`, доски внутри группы.  
- UI: `/work/groups`, `/work/groups/:id`.

Личные доски: по решению продукта — либо `WorkGroup` типа «Личное», либо `work_group=null` + owner-only (зафиксировать в задаче на реализацию).

## 7. Статусы задач и колонки (единая логика)

**Статус задачи** = колонка **`Column`**, в которой стоит карточка.  
У колонки есть **`semantic`** — значение из **единого перечня** (для отчётов и правил «завершено»). Пользовательские колонки без смысла для метрик → `semantic = null`.

### 7.1. Единый перечень семантики (`Column.semantic`)

| Значение | Типичные подписи (i18n) | Завершённость |
|----------|-------------------------|---------------|
| `planned` | Новая / Запланирована | нет |
| `in_progress` | В работе / В процессе | нет |
| `review` | На проверке / На контроле | нет |
| `paused` | На паузе | нет |
| `cancelled` | Отменена / Отклонена | **терминал (неуспех)** |
| `done` | Завершена / Закрыта | **терминал (успех)** |
| `backlog` | Бэклог | нет |
| `development` | В разработке | нет |
| `testing` | Тестирование / QA | нет |
| `released` | Готово к релизу / Выпущено | **терминал (успех)** |

**Отчёты «закрытые успешно»:** `semantic in ('done', 'released')`. **«Активные»:** не терминал и не `cancelled`.

### 7.2. Пресеты при создании доски

Пресет — только **начальный набор колонок** (порядок + `title` + `semantic`).

| Пресет | Колонки по умолчанию (порядок слева направо) |
|--------|-----------------------------------------------|
| **`generic_pm`** | `planned` → `in_progress` → `review` → `paused` → `done` → `cancelled` (порядок `paused`/`cancelled` можно поменять в настройках доски). |
| **`it_sdlc`** | `backlog` → `development` → `testing` → `released`. |
| **`custom`** | Пустая доска или одна колонка `planned`; дальше вручную. |

Можно **добавлять** колонки с любой семантикой из таблицы §7.1 (например, к `generic_pm` добавить `testing`).

### 7.3. Модель данных

| Сущность | Поля (суть) |
|----------|-------------|
| `Board` | `preset` (`generic_pm` \| `it_sdlc` \| `custom`), FK `work_group`. |
| `Column` | `board`, `title`, `position`, `semantic` (nullable, из §7.1), опционально `wip_limit`. |
| `Task` | `column`, `position`, исполнитель, сроки, связи на CRM-сущности (позже). |

## 8. API (сводка)

- **`GET/PATCH /api/v1/profiles/me/dashboard/`** — схема виджетов и раскладка для **`/dashboard`** (валидация типов виджетов по роли).  
- `/api/v1/tasks/work-groups/`, `/boards/`, `/columns/`, `/tasks/` — рабочий контур.  
- `/api/v1/work/dashboard/` (черновик) — агрегаты для **`/work`** (списки групп, досок, задач с дедлайнами — можно переиспользовать внутри виджетов).  
- `/api/v1/internal/...` — только `EmploymentKind.internal`.
