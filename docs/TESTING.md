# Тестирование: бэкенд и фронтенд

Тесты — **обязательная** часть качества: и **Django/DRF**, и **Quasar/Vue** должны проверяться в CI на каждый PR. Ниже — целевой набор инструментов и зон покрытия.

## 1. Принципы

- **Пирамида:** много быстрых **unit/интеграционных** тестов, меньше **E2E** (дороже в поддержке).  
- **Критичные пути** (auth, RBAC, оплата если появится) — покрытие в приоритете.  
- **Регрессии** ловятся CI: без зелёных тестов (и линтеров) merge не выполняется (branch protection).

## 2. Бэкенд (Django + DRF + Channels)

| Инструмент | Назначение |
|------------|------------|
| **pytest** + **pytest-django** | Запуск тестов, фикстуры БД, маркеры. |
| **pytest-cov** | Покрытие кода (отчёт в CI). |
| **factory_boy** | Фабрики `User`, `Board`, `Message` вместо ручного создания в каждом тесте. |
| **APIClient** / **pytest-drf** паттерны | HTTP к вьюхам: статусы, тело JSON, права доступа. |
| **channels.testing** / **pytest-asyncio** | Потребители WebSocket, `communicator` (по мере сложности). |

**Что тестировать в первую очередь:**

- регистрация, логин, refresh, logout (blacklist); капча после N ошибок;  
- доступ к стене/ЛС/**сообществам** (роль `user`) и к **рабочим группам** (роль `employee`, членство); разграничение **`internal`** / **`partner`** на API `internal/`;  
- сервисы канбана (перенос задачи, порядок колонок); **сид пресетов** колонок (`generic_pm`, `it_sdlc`) и поле `semantic`;  
- **`GET/PATCH .../me/dashboard/`** — валидация типов виджетов по роли (`user` не может сохранить `tasks_due` без прав и т.д.);  
- healthcheck и критичные сериализаторы.

База в CI: **PostgreSQL** (сервис в GitHub Actions), как в [CI-CD.md](./CI-CD.md); при желании локально SQLite только для быстрых прогонов — с осторожностью (фичи Postgres).

## 3. Фронтенд (Quasar + Vue 3 + Vite + TypeScript)

| Инструмент | Назначение |
|------------|------------|
| **Vitest** | Юнит-тесты и тесты компонентов (нативно дружит с Vite). |
| **@vue/test-utils** | Монтирование SFC, `mount`, события, props. |
| **happy-dom** или **jsdom** | Окружение DOM для Vitest. |
| **MSW** (Mock Service Worker) | Мок HTTP API в тестах без реального бэка. |
| **Playwright** или **Cypress** (фаза 2) | E2E: сценарии «вход → открыть чат» по мере стабилизации UI. |

**Что тестировать в первую очередь:**

- **composables** (`useAuth`, `useValidation`, `usePermissions`) — чистая логика;  
- **Pinia stores**;  
- ключевые **компоненты** (формы входа, guard маршрутов — через router mock);  
- по возможности снимки для статичных UI-блоков (не злоупотреблять).

## 4. CI

- Отдельные job’ы: `backend-test` (`pytest`), `frontend-test` (`npm run test` / `vitest run`).  
- Порог покрытия (например 60% на новый код) — включить позже через **codecov** или встроенный `--cov-fail-under` после набора базы тестов.

Подробнее команды — в [CI-CD.md](./CI-CD.md).
