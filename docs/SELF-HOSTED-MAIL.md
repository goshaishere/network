# Собственный почтовый сервер (опциональная экстра-фича)

**Транзакционные письма** (сброс пароля, уведомления) в MVP идут через **внешний SMTP** (SendGrid, SES, Mail.ru для бизнеса и т.д.) или в dev — **Mailhog**.

## Стоит ли «замутить» свой почтовый сервер?

**Короткий ответ:** для продакшена **полноценный почтовый сервер** (Postfix + Dovecot + антиспам + DKIM/SPF/DMARC + репутация IP) — **отдельный продукт** по сложности сопровождения. Для соцсети почти всегда выгоднее **transactional API** провайдера.

| Подход | Сложность | Когда имеет смысл |
|--------|-----------|-------------------|
| **API провайдера** (SES, SendGrid, …) | Низкая | Почти всегда для сброса пароля и рассылок |
| **Свой Postfix только на отправку** (relay) | Средняя | Строгие compliance, изоляция контура; всё равно нужны DNS и мониторинг репутации |
| **Полный почтовый домен** (входящая+исходящая) | Высокая | Если вы реально становитесь почтовым провайдером для пользователей |

## В пайплайне как экстра

В [PROJECT-PIPELINE.md](./PROJECT-PIPELINE.md) блок **«Экстра»**: собственный почтовый сервер не входит в MVP и не блокирует разработку; при появлении требования — отдельная задача/репозиторий (compose с Postfix для теста **или** полноценная почта под supervision DevOps).

**Рекомендация:** в основном репозитории Network — только **настройки SMTP** через env; сам почтовый стек не встраивать в критический путь MVP.
